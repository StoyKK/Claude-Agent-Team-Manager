## Component Inventory

Phase 1 modifies exactly one component and creates two new utility modules. No new visual components are introduced.

### Modified: `src/components/common/Toast.tsx`

**Change:** `ToastContainer` must render via `createPortal(…, document.body)` instead of inline in the React tree.

**Visual contract (locked per D-02):**
- Position: `fixed`, `bottom: 16px`, `right: 16px`
- Stack direction: `flexDirection: column`, `gap: 8px`
- z-index: `99999` (raised from current 10000 to clear wizard overlay at 20000)
- `pointerEvents: none` on container, `auto` on individual toast items
- Width: `maxWidth: 320px` per item
- Animation: slide-in from right on enter, slide-out to right on leave (existing keyframes unchanged)
- Colors: unchanged (see Color section)
- Duration: 3-second auto-dismiss (unchanged)

**Why createPortal:** The wizard overlay at `SetupWizard.tsx` lines 166–176 applies `backdropFilter: blur(8px)`. This creates a CSS stacking context. Any child z-index inside this context is relative to that context, not the document root. `createPortal` renders at `document.body`, completely outside the stacking context. A z-index bump alone does not work.

### New: `src/services/logger.ts`

No visual contract. Developer-facing utility only. Users never see the logger directly — they see structured log output in the developer console or Tauri log sink.

### New: `src/types/errors.ts`

No visual contract. Developer-facing typed error hierarchy. User-visible error surfaces exclusively via toast notifications (D-06).

## Copywriting Contract

Source: CONTEXT.md D-06, D-07, and REQUIREMENTS.md BUG-04.

| Element | Copy |
|---------|------|
| Primary CTA — enable teams | "Enable Agent Teams" |
| Primary CTA — complete wizard | "Get Started" |
| Toast: API key saved | "API key saved" |
| Toast: API key missing | "Please enter an API key" |
| Toast: teams enabled | "Agent teams enabled!" |
| Toast: teams enable failed | "Failed to enable teams" |
| Toast: wizard save failed (BUG-04) | "Could not complete setup — check that your home directory is writable" |
| Toast: browser open failed | "Could not open browser" |
| Toast: generic save failed | "Could not save settings. Please try again." |
| Toast: JSONC parse error (new) | "Could not read settings file — it may be malformed" |
| Toast: settings fallback (informational) | none — silent fallback, no toast needed |

Rules:
- User-facing copy is friendly and actionable (D-07). Technical details (file path, error code, stack trace) go to the structured logger only.
- Error toasts name the problem and imply what to do next. No raw JavaScript error messages shown to users.
- Success toasts are brief confirmations, no exclamation marks except "Agent teams enabled!" (pre-existing, preserved per D-02).

Destructive actions in Phase 1: none. No delete/reset/overwrite actions are introduced. The `handleEnableTeams` write-back modifies `~/.claude/settings.json` but is not destructive (it adds a key, not removes data, and preserves comments via `modify()` + `applyEdits()`). No confirmation dialog required.

## Interaction Contract

### Toast visibility (BUG-01)

- Toasts are visible at all times, including when the setup wizard overlay is active
- The wizard overlay (`zIndex: 20000`, `backdropFilter: blur(8px)`) must never obscure toasts
- Implementation: `createPortal(toastJsx, document.body)` in `ToastContainer`
- z-index of toast container: `99999`
- No changes to toast timing, animation, or appearance

### Wizard error feedback (BUG-04)

- When `handleGetStarted()` fails, the user sees a toast error message
- The message is: "Could not complete setup — check that your home directory is writable"
- The wizard remains open (does not close on error)
- The "Get Started" button does not show a loading/disabled state (no change to button behavior beyond what already exists)

### JSONC silent parsing (BUG-02, BUG-03)

- No visible UI change when JSONC parsing succeeds — the wizard proceeds normally
- If `jsonc-parser` encounters a malformed file it cannot recover: show toast "Could not read settings file — it may be malformed" (error type)
- The global `~/.claude/settings.json` fallback (BUG-03) is entirely silent — no indicator, no toast. The wizard proceeds normally if teams are enabled in the global file.

### Log output (LOG-01, LOG-02)

- No user-visible interaction. Log output appears in the developer console or the Tauri log viewer.
- Log format: `[ATM] [LEVEL] [context] message`
